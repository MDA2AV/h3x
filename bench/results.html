h3x: HTTP/3 load generators vs QUIC servers

Two load generators, five QUIC server stacks, two connection regimes. Neither client wins outright: the shape of the workload decides it. h3x owns few fat connections with heavy stream multiplexing; h2load owns many thin connections.

Each cell is one 10 second run fetching the same 1 KB file over HTTP/3. Loopback, single box, TLS 1.3. Bars are normalized to the fastest cell across all tables. Numbers are one representative run and vary about 10 percent between runs.

CPU split: each server is pinned to 6 performance cores (CPUs 0 to 11, 12 SMT threads, 12 worker threads). Both clients get the same other 20 threads (CPUs 12 to 31), verified identical, neither pinned per thread. Server and client never share a core.   Clients: h3x = quicly (UDP GRO, cross-connection send batching, and connection-establishment pacing on; send-batch is the request refill knob, swept below), h2load = ngtcp2.   Servers: h2o = quicly · nginx = OpenSSL-QUIC · haproxy = HAProxy · caddy = quic-go · envoy = Google quiche

Few connections, many streams: 8 connections x 32 streams

A browser-like shape: a handful of fat connections each multiplexing many concurrent streams (256 requests in flight). This is what h3x is built for. It wins on 4 of 5 servers, 0 failures anywhere, and beats h2load by nearly 6 times on h2o. The send-batch knob (accumulate this many freed stream slots before refilling, so quicly packs requests into fuller datagrams) adds a few percent on the client-bound servers and costs a few on the ones that batch their own responses.

Many connections, few streams: 512 and 1024 connections x 8 streams

A crowd-of-clients shape: hundreds of thin connections, each with only a few streams. Now the work is connection handling, not multiplexing, and it flips: h2load is faster on 4 of 5 servers and clean in every cell, while h3x develops a failure tail. Send-batch is left off here because it does not help this regime (see the curve below). h3x's throughput peaks near 512 connections and falls at 1024, the sign of over-subscription.

Why h3x fades as connections rise

h3x against h2o, holding streams at 8 and sweeping the connection count. Throughput falls with every connection added past 64, and it does so with 0 failures right up until 1024. A clean monotonic decline with no failures is the signature of per-connection overhead, not churn: each QUIC connection is independent work the event loop services every pass (its own pacing, loss and idle timers, congestion control, and per-connection encryption), so the cost grows with the connection count while the fixed 20 core budget stays the same. The failure tail is a separate, second effect that only appears once the loop is too busy to service every connection's timers in time, so some drop and re-handshake.

h2load's curve is flatter because its per-connection loop is leaner (ngtcp2, no httpclient connection-pool layer) and it holds a fixed connection set it never refills, so it does not churn. This iteration added connection-establishment pacing to h3x (each worker caps its concurrent handshakes and ramps the active set as connections come up); it roughly halved the h2o 1024-connection failure tail (about 3,700 to 2,000) with no throughput cost, but it does not change the decline. The decline is the fixed per-connection cost. And on the servers where h2load leads on throughput, the gap is the receive side: GRO only coalesces responses from servers that GSO-batch (h2o, haproxy), so against nginx quicly does more per-packet work than ngtcp2, and no transport knob tried (recv window, UDP payload size, ACK frequency) moved it. Closing that gap would mean not using quicly, the opposite of the reuse-h2o goal.

The receive side: UDP GRO

GRO is a build-time change in the h2o library h3x links, not a runtime flag, so it is on for every cell. It coalesces the many small response datagrams a server GSO-batches into one recvmmsg, and helps most against servers that batch their sends (h2o, haproxy) and roughly not at all against nginx, which does not. It never hurts, so it stays on.
